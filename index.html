This document presents a set of 54 metadata elements (organized into five categories) to usefully describe Web-based Application Programming Interfaces (APIs). These elements were developed by the Big Data to Knowledge (BD2K) API Interoperability Working Group, which conducted a survey of API metadata used in the real world. This group developed the smartAPI Specification as an extension of existing repositories such as Programmable Web, Biocatalogue, and available standards including Open API, schema.org, etc. The aim of the BD2K API Interoperability Working Group is to develop a strategy for maximizing interoperability and reuse of Web-based APIs. This specification aims to serve as a standard for API development that will facilitate the efficient communication among APIs and reduce development costs. The smartAPI Specification includes 21 metadata elements beyond those included in the Open API Initiative. The metadata elements are grouped into categories related to APIs, service providers, API operations, operation parameters, and operation responses. For each category, the metadata elements that are mandatory, recommended, or optional are described and illustrated by examples. The widespread adoption of the smartAPI Specification by the community promises to improve the efficiency and lower the costs of API development, promoting cross-API compatibility and resolving current challenges in API usage.

This document is a DRAFT that now includes a mapping to v2 and v3 of the OpenAPI specification. It correctly uses vendor extensions. Comments can be provided to the NIH CFWG API Mailing List.

Purpose of the smartAPI Specification

The purpose of the smartAPI Specification is to establish a set of standards for the description of metadata elements and values for the description of Application Programming Interfaces (APIs). This document was produced by the Big Data To Knowledge (BD2K) API Interoperability Working Group, one of the NIH Commons Framework Pilot Working Groups. The BD2K API Interoperability Working Group first conducted a survey of API metadata found in existing repositories and guidelines (see Section 2), and subsequently developed the smartAPI Specification as an extension of the Open API Initiative. The Open API Initiative aims to create, develop, and promote an API description format based on the Swagger Specification.

Survey of API Metadata in the Wild

This standard is the result of a survey by the BD2K API Interoperability Working Group of existing metadata repositories and specifications that describe APIs. The following eight resources were surveyed:

Repositories:
  • Biocatalogue, a registry of biological web services.
  • Programmable Web, a directory of internet-based APIs.
  • Tools and Data Services Registry, a registry with information about analytical tools and data services for bioinformatics.
Specifications:
  • Minimal Information About a Software, a key set of minimal fields can that provide maximum value when describing a software.
  • smartAPI specification, a specification that describes a semantically annotated web service that facilitates discovery and reuse of web-based API.
  • openAPI Initiative, created by a consortium of forward-looking industry experts who recognize the immense value of standardizing on how REST APIs are described.
  • Semantic Automated Discovery and Integration, a set of design patterns defining the behavior of data retrieval and/or analysis resources that must interoperate on the Semantic Web.
  • schema.org API Reference, a reference document for APIs as described by schema.org.

The smartAPI Specification includes 21 metadata elements beyond those included in the Open API Initiative. Examples of the 21 elements are the category to which the API belongs; metadata format and access mode at the API metadata level; the parameter type and parameter value type at the operation parameter level; and the conformance to a specified response profile at the operation response level.

We subsequently aggregated all the metadata elements from the eight resources to produce a common list of 54 API metadata elements. We subsequently divided these elements into five categories:

  • API Metadata: 20 elements
  • Service Provider Metadata: 6 elements
  • API Operation Metadata: 10 elements
  • Operation Parameter Metadata: 12 elements
  • Operation Response Metadata: 6 elements

Next, we discussed each of the metadata field amongst the working group members to re-evaluate its applicability and relevance and further classified them into whether it MUST, SHOULD, or MAY be included in the API description. The cardinality and datatype of metadata field were further specified along with a description and example. The results of the survey are available here. The key words "MUST", "SHOULD", and "MAY" in this document are to be interpreted as described in RFC 2119.

Audience

The target audience includes software developers, data scientists and informatics researchers, publishers of resource descriptions, and funding agencies.

API Metadata Elements

Name [MUST]

A human readable label for the API.

  • Cardinality: 1..1
  • Datatype: string
		info:

		  title: mygene.info

Description [SHOULD]

A human readable description of the service.

  • Cardinality: 0..1
  • Datatype: string
		info:

		  description: MyGene.info is a high performance API for gene descriptions.

Descriptors [MAY]

Keyword or concepts that describe the API.

  • Cardinality: 0..n
  • Datatype: string
 
        info:
          x-keywords:
            - genes
            - http://semanticscience.org/resource/SIO_010035

Version [MUST]

The version of the API. Specify API version using Semantic Versioning. The major.minor portion of the semver (for example 3.0) SHALL designate the feature set. Typically, .patch versions address errors in the API metadata document, not the feature set.

  • Cardinality: 1..1
  • Datatype: string
		info:
		  version: 2.0.0

Terms of Service [MUST]

A URL that describes the terms of use for the API.

  • Cardinality: 0..1
  • Datatype: URL
		info:

		  termsOfService: http://mygene.info/terms/

Access Point [MUST]

The base URL for the API upon which the path is added. Also include URL of mirrors that the API is available at.

  • Cardinality: 0..1
  • Datatype: URL
		server:

		  - url: http://mygene.info/v3/api
		    description: Production server
		  - url: https://mygene.info/v3/api
			description: Fictional HTTPS Production server
		  - url: http://mirror1.mygene.info/api
			description: Fictional mirror

Response Mime-Type [MUST]

A list of media types the APIs can produce. The media type definitions SHOULD be in compliance with RFC6838.

  • Cardinality: 0..n
  • Datatype: string
        components:

          responses:

            - content:

              - application/json:

Website [MAY]

A web page that describes the API.

  • Cardinality: 0..1
  • Datatype: URI
 
        info:

          x-homepage: http://mygene.info

Documentation [SHOULD]

Additional documentation for the API.

  • Cardinality: 0..n
  • Datatype: URL
		info:

		  documentation: http://docs.mygene.info/en/v3/

Developer Forum [MAY]

Forum or Mailing list.

  • Cardinality: 0..n
  • Datatype: URI
       info:
          x-developerForum: 
            - https://bitbucket.org/sulab/mygene.hub

Security [SHOULD]

Lists the required security schemes to execute this operation. The object can have multiple security schemes declared in it which are all required.

  • Cardinality: 0..1
  • Datatype: URI
		security:

		  - type: apikey
		    name: api_key
		    in: header

Access Restrictions [MAY]

Indicate whether there are restrictions to using the API. Values to use: none, limited, fee.

  • Cardinality: 0..1
  • Datatype: string
 
        info:

          x-accessRestrictions: 
           - none

Social Media [MAY]

Links to social media presence such as Twitter, Facebook, LinkedIn, GitHub, etc.

  • Cardinality: 0..n
  • Datatype: URI
 
        info:

          x-accounts:

            - name: twitter

              accountid: mygeneinfo

Location [MAY]

Location, city and country of the server hosting the API.

  • Cardinality: 0..1
  • Datatype: URI
 
         server:

           - url: http://mygene.info/v3/api

             x-location: USA

Implementation Language [MAY]

Language the API was written in.

  • Cardinality: 0..n
  • Datatype: string
 
        info:

          x-implementationLanguage: python

Maturity [MAY]

Maturity of the API. Values to use: development, production.

  • Cardinality: 0..1
  • Datatype: enum
 
        info:

          x-maturity: production

Publications [MAY]

Publications that describe the API.

  • Cardinality: 0..n
  • Datatype: URI
 
        info:

          x-publications: 
            - https://doi.org/10.1093/nar/gks1114

Funding [MAY]

Indicate sources of funding.

  • Cardinality: 0..n
  • Datatype: string
 
        info:

          x-funding:

            - name: National Institutes of Health
              x-id: https://www.wikidata.org/wiki/Q390551
              funding-id: HG008473

Service Provider Metadata

Responsible Organization [MUST]

Provide the name of the organization responsible for the service.

  • Cardinality: 1..n
  • Datatype: string
 
        info:

          contact:

            - name: The Scripps Research Institute
              x-id: https://www.wikidata.org/wiki/Q793867
              x-role: responsible organization
              url: http://www.scripps.edu/

Responsible Developer [MUST]

Provide the name, ID and email for the responsible developer.

  • Cardinality: 1..1
  • Datatype: URI
 
        info:

          contact:

            - name: Chunlei Wu
              x-id: https://orcid.org/0000-0002-2629-6124
              x-role: responsible developer
              email: cwu@scripps.edu

Contributor [MAY]

Indicate any other contributor to the conception, development, or support of the API.

  • Cardinality: 0..1
  • Datatype: string
 
        info:

          contact:

            - name: Andrew Su
              x-id: https://orcid.org/0000-0002-9859-4104
              x-role: contributor
              email: asu@scripps.edu

API Operation Metadata

Operation Identifier

A unique identifier for the operation.

  • Cardinality: 1..1
  • Datatype: string
 
        paths:
          - get:
             - 
          "operation title": "q"


Operation Description

Description of the operation.

  • Cardinality: 1..1
  • Datatype: string

OpenAPI element: operation.description

 
          "description": "Query string. Examples \"CDK2\", \"NM_052827\", \"204639_at\". The detailed query syntax can be found at http://docs.mygene.info/en/latest/doc/query_service.html"


Consumes

A list of MIME types the operation can consume.

  • Cardinality: 0..n
  • Datatype: string

OpenAPI element: consumes

 
          "consumes": "application/json"


http Method

The base path on which the API is served, which is relative to the host. Or a relative path to an individual endpoint. The field name MUST begin with a slash. The path is appended to the basePath in order to construct the full URL. The object describes the operations available on a single path. The different operations are"get, put, post, delete, options head, patch".

  • Cardinality: 0..1
  • Datatype: string

OpenAPI element: pathItem

 
          "http method": "get"


Authentication Mode

Lists the required security schemes to execute this operation. The object can have multiple security schemes declared in it which are all required.

  • Cardinality: 0..n
  • Datatype: string

OpenAPI element: securityDefinitions.basicAuth

 
          "authentication mode": "none"


Transfer Protocol

The transfer protocol of the API. Values MUST be from the list: "http", "https", "ws", "wss". This optional value will override the top-level schemes if present.

  • Cardinality: 0..1
  • Datatype: string0

OpenAPI element: schemes

 
          "transfer protocol": "http"


Keywords

Keywords describing the operation.

  • Cardinality: 0..n
  • Datatype: string

OpenAPI element: operation.tags

 
          "keywords": [ "query" ]


Path Template*

The path for the operation.

  • Cardinality: 0..n
  • Datatype: string

 
          "path template": "/query"


Operation ID

An identifier/URI for the operation.

  • Cardinality: 0..1
  • Datatype: string

OpenAPI element: operation.operationID

 
          "operation ID": "queryGene"


Operation Documentation

Documentation of the particular operation.

  • Cardinality: 0..n
  • Datatype: URI

OpenAPI element: operation.externalDocs

 
          "operation documentation": " http://docs.mygene.info"


Access Restrictions*

Access restrictions to invoke the operation.

  • Cardinality: 0..1
  • Datatype: string

 
          "access restrictions": "free"


Collection*

Names of collections of which the resource is a part e.g. a suite, library etc.

  • Cardinality: 0..n
  • Datatype: string

 
          "collection": "Gene Annotation"


Operation Parameter Metadata

Operation Parameter Name

Name of the operation parameter.

  • Cardinality: 1..1
  • Datatype: string

OpenAPI element: bodyParameter.patternProperties.name

 
          "operation parameter name": "q"


Operation Parameter Description

Description of operation parameter.

OpenAPI element: bodyParameter.patternProperties.description

  • Cardinality: 1..1
  • Datatype: string

 
          "description": "multiple query terms seperated by comma (also support \"+\" or white space), but no wildcard, e.g., \"q=1017,1018\" or \"q=CDK2+BTK\""


Location

Determines the location of the parameter.

  • Cardinality: 1..1
  • Datatype: string

OpenAPI element: bodyParameter.patternProperties.in

 
          "location": "query"


Parameter Type*

Type of the parameter in terms of its role in the operation.

  • Cardinality: 0..n
  • Datatype: string

 
          "parameterType": "inputParameter"


Parameter Value Type*

Semantic identifiers of particular parameters, retreived from identifiers.org

  • Cardinality: array of URIs
  • Datatype: string

 
          "parameter value type": "http://identifiers.org/ensembl"; "http://identifiers.org/ncbigene"


Min

Minimum occurrence of the parameter.

  • Cardinality: 0..n
  • Datatype: integer

OpenAPI element: minItems

 
          "min": "0"


Max

Maximum occurrence of the parameter.

  • Cardinality: 0..1
  • Datatype: integer

OpenAPI element: maxItems

 
          "max": "1"


Data Format

Input/Output data format.

  • Cardinality: 0..1
  • Datatype: string

OpenAPI element: type

 
          "format": "string"


Default Value

Default value for each input/output.

  • Cardinality: 0..1
  • Datatype: string

OpenAPI element: default

 
          "default value": "cdk2"


Example Value

Example value for each input/output.

  • Cardinality: 0..1
  • Datatype: string

OpenAPI element: examples

 
          "example value": "cdk2"


Operation Response Metadata

Response Format

A list of MIME types the APIs can consume. Here are some examples of MIME types.

  • Cardinality: 1..n
  • Datatype: string

OpenAPI element: mimeType

 
          "response format": "application/json"


Example Response Value

An example of the response value.

  • Cardinality: 0..n
  • Datatype: integer

OpenAPI element: response.examples

 
          "example response value": "200"


Response Header

A list of headers that are sent with the response.

  • Cardinality: 0..n
  • Datatype: object

OpenAPI element: response.headers.hearders.header

 
          "response header": "application/json"


Response Code

A container for the expected responses of an operation. The container maps a HTTP response code to the expected response. The list of possible responses as they are returned from executing this operation.

  • Cardinality: 0..1
  • Datatype: integer

OpenAPI element: response.patternProperties

 
          "response code": 200


Response Description

A description of the response.

  • Cardinality: 0..1
  • Datatype: string

OpenAPI element: responseValue.response.description

 
          "response description": "Successful response"


Response Profile*

Conformance to a specified format.

  • Cardinality: 0..n
  • Datatype: URI

 
          "response profile": "http://www.w3.org/ns/anno.jsonld"


Response Data Type*

Semantic annotation of the response using the smartAPI profiler.

  • Cardinality: 0..n
  • Datatype: array of objects

 
          "responseDataType": [{"elementPath":"ec","datatype":"http://identifiers.org/ec-code/"},{"elementPath":"ensembl.gene","datatype":"http://identifiers.org/ensembl/"}]
